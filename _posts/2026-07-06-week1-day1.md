# Jit Compilation
- requires array shapes to be static & known at compile time

```
def get_negatives(x):
  return x[x < 0]

jit(get_negatives)(x)
> NonConcreteBooleanIndexError: Array boolean indices must be concrete; got bool[10]
```

But we can make it a fixed size array by using jnp.where() which lets us jit compile

```
def get_negatives(x):
  return jnp.where(x > 0, x, 0)

jit(get_negatives)(x)
```

# jax.vmap
- interesting you can just add a @jax decorator to a function rather than manully writing jax.jit()

Seems like all methods can be written for a single operation and made "batch-size aware" through vmap. You can also specify which dimension you want to iterate over:

- With 1 argument: jax.vmap(fn, in_axes=(0))
- With 2 arguments: jax.vmap(fn, in_axes=(0, 1))
- 1 argument held constant: jax.vmap(fn, in_axes=(0, None))
- By name: jax.vmap(fn, in_axes=({"arg1": None, "arg2": 1})
- Output argument: jax.vmap(fn, out_axes=(1))

A good mental model might be to think of it as a slice -> function call -> stack

# Pseudorandom Numbers
- takes a key that has to be split to generate new random numbers (thread-safe)
- random funcitons do not modify the key (don't reuse keys unless it's intended)

```
new_key, subkey = random.split(key)
```
So it generates 2 new keys every split and is deterministic for a given key.

# jax.debug.print
- use this instead of python print in jitted methods

```
import jax
jax.config.update("jax_debug_nans", True)
jax.config.update("jax_disable_jit", True)

or

with jax.debug_nans(True):
    train_step(state)
```
Useful debugging tips, especially the debug nans for loss function/gradient computation

A few more that might be useful:
- jax.config.update("jax_log_compiles", True) # logs whenever JAX recompiles jitted function
- jax.config.update("jax_debug_infs", True)   # for infs, -infs

# Notes
Having pure functions and immutable state makes code tracing/compilation much easier to have and debug. A way of detecting "impurity" is to ask yourself, if I called this method twice with the same inputs, would it produce different outputs? (ex. relies on global data, random seed, mutating arguments, etc)
