`arr.at[0].set(5)` doesn't mutate `arr` — returns a new array. Rewrote three NumPy snippets (index assign, increment-in-place, boolean masking) using `.at[]`. Felt clunky at first, especially the boolean masking one, had to look up `.at[mask].set(val)` syntax twice before it stuck.

Question I still have: is there a performance cost to constantly allocating "new" arrays instead of mutating? Need to check if XLA can optimize this away under jit.
